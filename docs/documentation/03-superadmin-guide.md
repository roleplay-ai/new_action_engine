# Superadmin Guide (role: `superadmin`)

> Part of the Nudgeable Action Engine documentation set. See [README.md](./README.md) for the index and [00-architecture-overview.md](./00-architecture-overview.md) for cross-cutting concepts. This document is technical + feature-level.

---

## 1. Access & how a superadmin is created

Route: `/superadmin`. Gate (repeated verbatim across every superadmin page and server action rather than a single shared helper):

```
allowed if profiles.role === 'superadmin'
       OR user.email === SUPERADMIN_EMAIL   (env var, default "admin@actionengine")
```

The email fallback is a deliberate anti-lockout mechanism, not a bug — it covers the case where the known superadmin's `profiles.role` hasn't synced yet.

**There is no in-app way to create a superadmin.** The only superadmin account comes from a database seed migration (fixed UUID, email `admin@actionengine`, seeded password `admin@actionengine` — **change this in any real deployment**). Promoting an existing user to `superadmin` isn't exposed anywhere in the UI either — the user-edit action explicitly only accepts `role: "user" | "admin"`. A second superadmin can currently only be created by direct database access.

### Navigation

Three tabs: **Companies** (`/superadmin`), **Users** (`/superadmin/users`), **Emails** (`/superadmin/emails`). There is no global/cross-company analytics tab — analytics exist only inside the [company admin panel](./02-admin-guide.md#7-analytics), scoped per company.

## 2. Company management

Route: `/superadmin` (default). A simple list — name, slug, created date — with inline edit (pencil icon → edit name/slug → save). Creating a company just needs a name (slug optional). **There is no delete-company action.** No per-company settings, logo, or user count shown in this list.

---

## 3. Global user management

Route: `/superadmin/users`. A flat table of **every** user across **every** company (up to 500 — the underlying Supabase Auth listing call is capped, there's no pagination beyond that in the UI).

- **Create user** — email, password (chosen by the superadmin, not auto-generated by Supabase), full name, company, and role (`user` or `admin`). The account is created pre-confirmed (no email verification step). As with company-admin-created users, the plaintext password is stored server-side so it can be emailed later (see §5).
- **Edit user** — a single consolidated modal handles renaming, reassigning company, and changing role between `user`/`admin`. Assigning `role: "admin"` requires picking a company (a company admin must belong to a company). Editing a superadmin target is special-cased — only their name can be changed; their company/role fields are locked.
- **Delete user** — hard-deletes the Supabase Auth account (cascades to their `profiles` row and everything hanging off it). Blocked for self-deletion and for any other superadmin account.
- **Rotate auto-login key** — invalidates a user's existing permanent login link by generating a new key (useful if a link leaked).

This is how **assigning a company admin** actually happens today: pick a user in this table, open Edit, set role to `admin` and pick their company. There is no per-company "promote one of these users" view — it's managed from the flat global list.

> **Note on unused legacy actions**: the codebase also contains standalone `assignUserToCompany`, `removeUserFromCompany`, `setCompanyAdmin`, and `removeCompanyAdmin` functions matching an earlier design. They still work if called directly, but **no current UI button calls them** — everything in this section is actually driven by the single `updateUserBySuperadmin` action. Don't expect to find dedicated "assign to company" / "make admin" buttons in the UI; it's all in the one Edit modal.

## 4. Auto-login — testing & sending

Every user profile carries a permanent, non-expiring `persistent_login_key`. Visiting `{app}/api/auto-login?key=<that key>` logs that specific user in immediately — no password prompt. Two panels on `/superadmin/emails` deal with this, plus a third entry point on the Users page:

- **Auto-Login Testing Panel** — an internal/dev tool. Lists every user with a key and gives you Copy Link, Test (opens the link in a new tab so you can confirm the flow works, e.g. in an incognito window), and Rotate (invalidate + regenerate) actions. **Sends no email** — it's purely for verifying the mechanism works.
- **Auto-Login Email Panel** — select users, click "Send login email," and each gets an actual Resend email containing their personal auto-login link. This is the real end-user-facing delivery mechanism.
- The Users page also has its own "Send login email" toolbar wired to the same send action — so this capability is reachable from two different screens.

### How the link actually resolves

Clicking `{app}/api/auto-login?key=<uuid>` looks the key up (rejecting anything malformed), finds the user's email, and asks Supabase to generate a genuine short-lived magic-link token on the fly — the permanent key itself is never treated as a login credential by Supabase, only as a lookup index. The browser is then redirected through Supabase's own verification step, lands on `/auth/callback`, and the app establishes a normal session from there. Every attempt — successful or not, including bad/expired keys — is written to an internal audit log (IP, user agent, success flag) that isn't currently surfaced in any UI; reviewing it would require direct database access.

## 5. Credential delivery

Distinct from auto-login: this is for users who need their **actual password** (for password-based login rather than the one-click link). When any user is created (by a company admin or superadmin), the plaintext password they were given is stored server-side (Supabase Auth itself only keeps a hash, so this is the only place it's recoverable afterward). Two things read from that stored password:

1. **Auto-login emails** (superadmin, this panel) — do *not* include the password, only the one-click link.
2. **Credential emails** (company admin's Email Management screen, see [02-admin-guide.md](./02-admin-guide.md#6-email-management)) — *do* include the plaintext password plus a link to the normal `/login` page.

Stored passwords are never auto-deleted, specifically so credentials can be resent if a user loses the original email. Treat this table as sensitive — it's one of the few tables with no client access at all (service-role only), by design.

## 6. Email scheduling

A recurring-send configuration system, superadmin-only, on `/superadmin/emails`.

A **schedule** has: a name, an email template (a fixed key from `lib/email-templates.ts` — "Weekly Challenges" or "Login Credentials"), a list of recipient user IDs, a cadence type, and a run time. Supported cadence types in the underlying logic: daily, weekly, every N days, or a one-time specific date — **but the create form in the UI currently only offers Weekly or Specific Date**; daily/every-N-days exist in the backend and database constraint but aren't reachable from this screen today.

- **Create** — computes the first run time and saves the schedule as active.
- **Toggle** — pause/resume without deleting.
- **Delete** — removes it entirely.
- **Run now** — manually fires the same send logic immediately, useful for testing without waiting for the next scheduled run (documented in the code as a workaround for Vercel's free-tier cron only running once a day).
- Editing recipients after creation is implemented in the backend but **not wired to any button** in this panel — once created, a schedule's recipient list isn't editable from the UI.

Every send — scheduled or manually triggered — writes a per-recipient audit row to an email log table (status, error if any, who triggered it). That log isn't currently rendered anywhere in the UI (the read function exists but no screen calls it) — reviewing send history requires direct database access today.

## 7. Cron / automated sending

Exactly **one** automated job runs in production: a daily email-scheduler cron (configured in `vercel.json`), which checks for any due schedules from §6 and sends them, then advances each schedule's next-run time.

> **Known inconsistency worth flagging**: both the in-app UI copy and a code comment describe this cron running on a different cadence than it actually does (one says hourly, one says a specific evening time). The schedule actually configured in the deployment is **once daily, 04:30 UTC (10:00 AM IST)**. If you're debugging "why didn't my scheduled email send on time," trust the deployment config, not the UI text.

A second cron route for "package activation" exists in the codebase but does nothing (an explicit no-op) and isn't even registered to run — package delivery timing is entirely computed on-the-fly wherever a user's or admin's screen queries it, not by any background job. See [00-architecture-overview.md](./00-architecture-overview.md#4-high-level-data-flow).

## 8. Things to know that aren't obvious from the UI

- The service-role Supabase key is a hard dependency for the entire superadmin panel — if it's not configured, both the Users and Emails screens show a warning banner and the global user list/email features simply don't work.
- Self-protection is built in: a superadmin can't delete their own account, can't delete another superadmin, and can't edit another superadmin's company/role (only their name).
- There's no global/cross-company analytics view — if you need company-level metrics, use that company's [Admin Dashboard](./02-admin-guide.md#2-dashboard) (superadmin can access any company's admin panel via the company picker there).
